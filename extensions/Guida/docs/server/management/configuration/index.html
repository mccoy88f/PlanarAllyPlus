#Configurazione del server

Modifiche recenti

La configurazione del server è stata completamente modificata nella release 2025.2. La config descritta in questo documento si riferisce al nuovo formato.

Di default NON è presente alcun file di configurazione. PA usa invece valori predefiniti per tutte le impostazioni. I valori predefiniti della config si vedono qui

Per cambiare il comportamento, crea un file config.toml nella cartella data del server.

Quando fornisci un file di config parziale, i campi/sezioni mancanti verranno sostituiti con i valori predefiniti dove rilevante.

#Formato

Come indica l’estensione, la config usa il formato TOML.

La config è suddivisa in sezioni più piccole con intestazione [section], e le sottosezioni sono indicate da [section.some.subsection]. Ogni (sotto)sezione è seguita da righe separate per configurare valori specifici.

Informazioni dettagliate sul funzionamento del formato TOML esulano da questo documento, ma consultando la config predefinita dovresti capire bene come funziona.

#Config predefinita

[general]
save_file = "data/planar.sqlite"
allow_signups = true
enable_export = true

[assets]
max_single_asset_size_in_bytes = 0
max_total_asset_size_in_bytes = 0

[webserver]
max_upload_size_in_bytes = 10485760

[webserver.connection]
type = "hostport"
host = "0.0.0.0"
port = 8000

[[logging]]
mode = "stdout"
level = "INFO"
Percorsi dei file

Quando specifichi percorsi di file o cartelle puoi usare percorsi assoluti (es. /home/user/example.txt) o relativi (es. data/planar.sqlite). Con un percorso relativo, sarà relativo alla cartella principale del server e NON alla cartella dove si trova la config.

#Impostazioni importanti

Alcune impostazioni che probabilmente vorrai controllare:

  • Config file di salvataggio
  • Utente admin
  • L’intera sottosezione webserver.connection
  • CORS

L’URL del client è un’altra che di solito conviene configurare, ma non è critica.

#Sezioni

Di seguito sono elencate tutte le opzioni possibili con i relativi valori predefiniti. Se queste informazioni diventassero obsolete e causassero errori, la fonte ufficiale è disponibile su GitHub in forma di codice.

#General

Nome sezione: general

#Campo: save_file

Predefinito: "data/planar.sqlite"
Tipo: string

Percorso del file di salvataggio. È un’impostazione cruciale: modificarla avrà un grosso impatto. Il file di salvataggio dovrebbe essere sottoposto a backup adeguato1.

Se il server si avvia e il percorso scelto non esiste, il server creerà un nuovo file di salvataggio.

Formato file di salvataggio

Nota che, al momento, PlanarAlly supporta solo il salvataggio in formato SQLite.

Attenzione

Mentre il server è in esecuzione potrebbero essere creati file aggiuntivi del database (es. planar.sqlite-shm e planar.sqlite-wal). Il file planar.sqlite-wal (write-ahead-log) è essenziale per il database di salvataggio; i dati andrebbero persi se non viene salvato. Il file planar.sqlite-shm (shared memory) non è critico e non serve farlo in backup.

Se il server si spegne correttamente questi file scompaiono. Possono però restare se il server non è riuscito a spegnersi in modo pulito. Quando copi un file di salvataggio, assicurati di copiare anche questi file se esistono, altrimenti il salvataggio potrebbe essere corrotto o non aggiornato.

#Campo: admin_user

Predefinito: N/D - campo opzionale
Tipo: string

Quando qui è specificato un nome utente, questo utente sarà considerato l’admin principale. Avrà accesso a funzionalità extra nel cruscotto per gestire utenti e campagne sul server.

#Campo: client_url

Predefinito: N/D - campo opzionale
Tipo: string
È generalmente consigliato configurare questo campo, ma non è richiesto per una configurazione minima.

Se impostato deve essere l’URL base completo dell’applicazione frontend (es. “https://app.planarally.io”).

Viene usato nell’URL di invito e nelle mail di recupero password. Se manca, l’URL di invito userà l’URL corrente del browser quando viene generato. Gli URL nelle email però non conterranno l’URL base perché il server non può prevederlo.

#Campo: allow_signups

Predefinito: true
Tipo: boolean

Questa impostazione stabilisce se gli utenti possono registrare un nuovo account tramite il frontend. Quando disabilitato la creazione di account è possibile solo tramite manipolazione del database.

#Campo: enable_export

Predefinito: true
Tipo: boolean

Quando questa impostazione è attiva è disponibile l’esportazione e importazione degli account per gli utenti.

È generalmente consigliata perché permette agli utenti di spostare i propri dati su un altro server e fornirmi dati per il debugging.

#Webserver

Nome sezione: webserver

Una delle sezioni più importanti da configurare è il webserver. Include l’importante sottosezione connection, che vorrai configurare per decidere come il server PA è collegato all’esterno.

#Sottosezione: connection

È una sottosezione (vedi la Config predefinita per un esempio di configurazione).

Configura come il server è connesso allo stack di rete. Deve essere una configurazione HostPort oppure socket. Di default si usa la combinazione HostPort.

#campo: type

Predefinito: "hostport"
Tipo: "hostport" o "socket"

Il campo type determina quale tipo di connessione viene usato.

# O si configura una connessione hostport
[webserver.connection]
type = "hostport"
...

# O una connessione socket
[webserver.connection]
type = "socket"
...

#Campi HostPort

#campo: host

Predefinito: "0.0.0.0"
Tipo: string

L’IP su cui il server tenterà di fare il bind.

#campo: port

Predefinito: 8000
Tipo: integer

La porta su cui il server tenterà di fare il bind.

Assicurati che la porta usata non sia occupata da altre applicazioni. Inoltre, assicurati che la porta sia accessibile dall’esterno della macchina. Specialmente in una rete privata (es. a casa) potrebbe essere necessario abilitare il port forwarding nella configurazione del router.

#Esempio config HostPort

[webserver.connection]
type = "hostport"
host = "192.168.0.1"
port = "80"

#Campi Socket

#campo: socket

Predefinito: N/D
Tipo: string

Il percorso del socket che verrà usato per il bind.

#Esempio config Socket

[webserver.connection]
type = "socket"
socket = "/var/run/pa.sock"

#Sottosezione: ssl

È una sottosezione opzionale; se omessa non viene applicata alcuna config SSL al server PA. Questo non impedisce di configurare SSL più in alto nella catena (es. nel reverse proxy).

È fortemente consigliato assicurarsi che SSL sia configurato da qualche parte nella catena!

#Campo: enabled

Predefinito: false
Tipo: boolean

Permette di attivare/disattivare la config SSL senza dover rimuovere completamente la sezione SSL o commentarla. Solo quando questo valore è true la config SSL verrà effettivamente applicata.

#Campo: fullchain

Predefinito: N/D
Tipo: string

Percorso del certificato full chain.

#Campo: privkey

Predefinito: N/D
Tipo: string

Percorso della chiave privata del certificato.

#Esempio config SSL

[webserver.ssl]
enabled = true
fullchain = "/etc/letsencrypt/live/planarally/fullchain.pem"
privkey = "/etc/letsencrypt/live/planarally/privkey.pem"

#Campo: cors_allowed_origins

Predefinito: N/D (campo opzionale)
Tipo: string / Lista di stringhe

Configurazione CORS per il server websocket.

Questo valore viene passato al server socketio; più info sono nella documentazione socketio. Si può impostare '*' per permettere tutte le origini, utile per test ma non sicuro.

Spesso va configurato con il valore wildcard (*) o con il dominio specifico corretto per far funzionare il tutto! La console del server dovrebbe loggare quando la connessione è stata rifiutata per CORS.

#Campo: max_upload_size_in_bytes

Predefinito: 10_586_760 (10 * 1024 ** 2 == 10 MB)
Tipo: integer

La dimensione massima di un singolo upload di dati dal client.

Va notato che agisce soprattutto come strategia di chunking. Gli upload degli asset e i trasferimenti delle campagne verranno spezzati secondo questo massimo, ma la dimensione totale può superare il limite! Vedi la sezione assets della config per info su come limitare la dimensione degli asset degli utenti.

#Mail

Nome sezione: mail

È una sezione opzionale e di default non è abilitata.

Al momento è usata solo per inviare email come parte del flusso di recupero password.

#Campo: enabled

Predefinito: false
Tipo: boolean

Permette di attivare/disattivare la config mail senza dover rimuovere completamente la sezione o commentarla. Solo quando questo valore è true la config mail verrà effettivamente applicata.

#Campo: host

Predefinito: N/D
Tipo: string

L’host del server SMTP, deve essere compilato.

#Campo: port

Predefinito: N/D
Tipo: integer

La porta del server SMTP, deve essere compilata.

#Campo: username

Predefinito: N/D (opzionale)
Tipo: string

Se previsto, si può specificare il nome utente per l’autenticazione.

#Campo: password

Predefinito: N/D (opzionale)
Tipo: string

Se previsto, si può specificare la password per l’autenticazione.

#Campo: default_from_address

Predefinito: N/D
Tipo: email-string

L’indirizzo email predefinito da usare per l’invio delle mail.

#Campo: ssl_mode

Predefinito: "starttls"
Tipo: Literal["ssl", "tls", "starttls", "lmtp"]
Aggiunto: v2025.2.1

Configura la modalità di connessione del client usata. Di default si usa il metodo starttls che dovrebbe coprire una vasta gamma di casi.

A volte potrebbe non funzionare correttamente; in quel caso puoi provare la modalità "ssl" o una delle altre.

#Stats

Nome sezione: stats

È una sezione opzionale che se non specificata userà un set di valori predefiniti. Controlla la raccolta e la trasmissione di statistiche d’uso anonime.

Vedi questa PR su GitHub per informazioni su cosa e perché.

#Campo: enabled

Predefinito: true
Tipo: boolean

Permette di attivare/disattivare la config stats senza dover rimuovere completamente la sezione o commentarla. Se impostato a false non verrà effettuata alcuna raccolta di statistiche.

#Campo: enable_export

Predefinito: true
Tipo: boolean

Permette di disabilitare l’esportazione delle statistiche al server stats di PA. La raccolta locale verrà comunque effettuata.

#Campo: export_frequency_in_seconds

Predefinito: 24 * 60 * 60 (1 giorno)
Tipo: int

Determina con quale frequenza le statistiche vengono inviate al server stats di PA.

Non è consigliato impostare un valore inferiore a 1 giorno. Frequenze consigliate: 1 giorno, 1 settimana o 1 mese.

#Campo: stats_url

Predefinito: https://stats.planarally.io
Tipo: string

L’URL di destinazione dei dati delle statistiche.

#Logging

Nome sezione: logging

È una sezione opzionale e la sua configurazione predefinita è loggare su standard output al livello “INFO”.

Più voci

La sezione Logging può essere specificata più volte per configurare più file di output. Come tale devi usare l’intestazione di sezione [[logging]]

#Campo level

Predefinito: INFO
tipo: Literal["DEBUG", "INFO", "WARNING", "ERROR", "CRITICAL"]

Il livello a cui questo logger scriverà. Anche i livelli superiori vengono loggati a questo livello. Ad esempio, se imposti ERROR verranno loggati sia ERROR che CRITICAL.

#Campo mode

Tipo: Literal["stdout", "file"]

Se specifichi stdout l’unico altro campo disponibile è level.

Ecco un esempio di logging che imposta l’output standard su critical:

...
[[logging]]
mode = "stdout"
level = "CRITICAL"
...

#Campo file_path

Predefinito: data/planarally.log
Tipo: string

Imposta il percorso di output per un log su file. Con Docker è consigliato usare la cartella data così è disponibile fuori dal container.

#Campo max_log_size_in_bytes

Predefinito: 200_000
Tipo: int

La dimensione massima di un file di log prima della rotazione al file successivo.

#Campo max_log_backups

Predefinito: 5
Tipo: int

Il numero di file di backup da mantenere; i file più vecchi verranno eliminati automaticamente.

Ecco un esempio per configurare un file logger con solo 2 file di backup. Può essere combinato con l’esempio dell’output standard per scrivere sia su standard out che su file.

...
[[logging]]
mode = "file"
level = "INFO"
max_log_backups = 2
...
